# 实验工作流 · 通用版 v2.0

> 适用于任何需要系统性设计、执行和记录的实验研究。
> 核心理念：**AI 辅助推进，用户始终主导。每个阶段用户必须真正理解内容后才能进入下一阶段。**

---

## 设计原则

- **AI 不替用户做决定**：AI 提供选项、分析利弊，最终选择由用户做
- **每个阶段必须有用户确认**：不允许 AI 自动跳入下一阶段
- **追问优先于填模板**：用户不需要一次性提供完整信息，AI 通过追问逐步补全
- **理解检验**：每个阶段结束前，AI 会用自己的话复述当前方案，用户确认理解无误后才推进

---

## 阶段总览

| 阶段 | 名称 | 核心任务 | 用户参与方式 | 产出物 |
|------|------|---------|------------|-------|
| 1 | 需求定义 | 把模糊想法澄清为结构化需求 | 多轮追问 + 最终确认 | 需求文档 |
| 2 | 方案设计 | 设计具体实验方案 | 逐项讨论 + 用户拍板 | 实验方案 |
| 3 | 方案评审 | 挑毛病、补细节 | 逐条审阅问题 + 决定是否修改 | 完善后的方案 |
| 4 | 规划拆解 | 拆节点、设参数表、排顺序 | 确认节点划分 + 审核参数表 | 实验手册 + 参数记录表 |
| 5 | 执行记录 | 按计划做、实时填结果 | 每个节点前理解目标、完成后确认结果 | 实验日志 |
| 6 | 复盘归档 | 分析结论、总结经验 | 确认结论 + 补充主观感受 | 复盘报告 |

---

## 阶段 1：需求定义

**目标**：通过多轮对话，把模糊的想法逐步澄清为结构化的实验需求。

**交互模式**：用户给出初步想法 → AI 追问 → 用户回答 → AI 归纳 → 循环，直到用户确认"可以了"。

**退出条件**：用户明确说"可以进入下一阶段"或"需求没问题了"。AI 不会主动跳过此阶段。

---

### Step 1-A：启动

用户只需提供最粗糙的想法，一句话也行。

```
【阶段1 · 启动】

我的领域/方向：___

我的初步想法：___
```

AI 收到后会：
1. 用自己的话复述它理解的你的想法（让你确认理解是否正确）
2. 识别最关键的模糊点
3. 每次只追问 1-2 个问题，不会一次甩出一堆

---

### Step 1-B：持续追问轮

正常回答 AI 的问题即可，不需要特殊格式。

遇到不确定的情况：
- "这个我还没想好" → AI 会列出可能的选项帮你思考
- "这个不重要" → AI 记录并跳过
- "你觉得呢" → AI 给出建议，但明确说明最终由你决定

---

### Step 1-C：需求确认

对话充分后，触发以下 Prompt 生成需求文档：

```
【阶段1 · 确认】

请根据我们的对话整理需求文档，包含：
1. 研究问题：这个实验要解决什么问题
2. 研究动机：现有方法/研究的不足是什么
3. 数据/材料：使用什么数据，来源是什么
4. 拟用方法：计划用哪些方法/模型
5. 评估标准：怎么判断实验成功还是失败
6. 边界说明：这个实验不打算解决什么

整理完后：
- 用一段话向我解释这份需求文档的核心逻辑，让我确认你的理解是否正确
- 列出还存在歧义或不确定的地方
- 等待我确认后再进入下一阶段
```

---

## 阶段 2：方案设计

**目标**：基于需求文档，和用户一起设计具体可执行的实验方案。

**交互模式**：AI 提出方案草稿 → 逐项和用户讨论 → 用户拍板每个关键决策 → 最终确认。

**退出条件**：用户逐项确认方案内容，并明确说"方案没问题"。

---

### Step 2-A：方案草稿

```
【阶段2 · 方案草稿】

需求文档如下：
[粘贴阶段1产出的需求文档]

请帮我起草一份实验方案，包含：
1. 实验目标
2. 方法选择（每个方法说明为什么选它）
3. 对照组/基线设置
4. 评估指标
5. 数据处理流程

注意：
- 每个有多种选择的地方，列出选项和利弊，不要直接替我决定
- 草稿完成后，逐项问我是否认可，有异议的地方一起讨论
```

---

### Step 2-B：逐项确认

AI 会针对方案中每个关键决策点单独询问，例如：

> "基线方法我建议用 X，原因是 ___。你认可这个选择吗？还是有其他想法？"

用户可以：
- 认可 → AI 记录并继续下一项
- 修改 → AI 更新方案并确认修改后的版本
- 不确定 → AI 进一步解释或提供对比

---

### Step 2-C：方案确认

```
【阶段2 · 确认】

请将我们讨论后的方案整理成最终版本。
整理完后，用通俗的语言向我解释这个方案的整体逻辑：
- 我们要比较什么？
- 怎么比？
- 用什么标准判断哪个更好？

等我确认理解无误后再进入下一阶段。
```

---

## 阶段 3：方案评审

**目标**：批判性审查方案，让用户亲自判断每个问题是否需要修改。

**交互模式**：AI 逐条提出问题 → 用户决定是否修改 → AI 更新方案 → 用户最终确认。

**退出条件**：用户审阅所有问题并做出决定，明确说"评审通过"。

---

### Step 3-A：评审

```
【阶段3 · 评审】

实验方案如下：
[粘贴阶段2产出的方案]

请从以下维度评审，每个问题单独列出：
1. 实验设计：对照组是否充分？对比是否公平？
2. 变量控制：有哪些潜在混淆因素？
3. 评估指标：是否能真实反映研究目标？
4. 可行性：数据/资源是否可获取？
5. 遗漏检查：有没有明显遗漏的实验或验证步骤？

注意：
- 每个问题都要附上具体的改进建议
- 逐条列出，每条结束后询问我是否需要修改，等我回复后再继续下一条
- 不要一次性输出所有问题
```

---

### Step 3-B：修改确认

每条问题处理完后：

```
【阶段3 · 修改确认】

请将所有修改整合进方案，输出最终版本。
然后给这个方案打分（1-10分），说明理由。
最后问我：是否认可这个评分和最终方案？
```

---

## 阶段 4：规划拆解

**目标**：将方案拆解为可独立执行的节点，用户确认每个节点的划分是否合理。

**交互模式**：AI 提出拆解草案 → 用户确认节点划分 → AI 设计参数表 → 用户审核字段 → 最终确认。

**退出条件**：用户确认节点划分和参数表字段，明确说"可以开始执行了"。

---

### Step 4-A：节点拆解

```
【阶段4 · 节点拆解】

最终方案如下：
[粘贴阶段3产出的方案]

请将实验拆解为可独立执行的节点，每个节点包含：
- 节点ID
- 目标（一句话）
- 关键参数
- 依赖关系（需要哪个节点完成后才能做）
- 优先级（核心 / 可选）

拆解完后，用通俗的语言解释：
- 为什么这样划分？
- 执行顺序是什么？
- 哪些节点最关键？

然后问我：这个划分是否合理？有没有需要合并、拆分或删除的节点？
```

---

### Step 4-B：参数表设计

用户确认节点划分后：

```
【阶段4 · 参数表设计】

请为每个节点设计参数记录表的字段，包含：
- 必填字段：节点ID、日期、参数配置、结果指标、状态（待执行/进行中/完成/跳过/失败）
- 建议字段：根据这个实验的特点补充

设计完后，解释每个字段的用途，让我判断是否有遗漏或多余的字段。
等我确认后生成最终版参数记录表。
```

---

## 阶段 5：执行记录

**目标**：按计划执行实验，每个节点前理解目标，完成后确认结果含义。

**交互模式**：节点前理解确认 → 执行 → 结果记录与解读 → 用户确认后进入下一节点。

**退出条件**：所有核心节点完成，用户确认参数表填写完整。

---

### Step 5-A：节点前确认

每个节点开始前：

```
【阶段5 · 节点前确认】

我准备开始执行节点：[节点ID] - [节点名称]

请用通俗的语言告诉我：
1. 这个节点要做什么，为什么要做
2. 我需要准备什么
3. 这个节点完成后，我应该得到什么结果
4. 如果结果和预期不符，可能是什么原因

我确认理解后再开始执行。
```

---

### Step 5-B：结果记录

节点执行完成后：

```
【阶段5 · 结果记录】

节点：[节点ID] - [节点名称]

本次结果：
- 实际参数配置：___
- 结果数据：[粘贴数值/输出]
- 观察到的现象：___

请帮我：
1. 将结果整理为参数表对应行的格式
2. 用通俗的语言解释这个结果说明了什么
3. 判断是否符合预期，如果不符合说明可能原因
4. 建议下一步执行哪个节点，并说明理由

等我确认理解后再继续。
```

---

### Step 5-C：异常处理

```
【阶段5 · 异常处理】

节点：[节点ID] - [节点名称]
问题描述：___
报错信息（如有）：[粘贴报错]
已尝试的解决方法：___

请帮我：
1. 诊断问题原因（用通俗语言解释）
2. 给出解决方案，优先不改变实验设计的方案
3. 如果必须调整实验设计，说明会影响哪些其他节点，让我决定是否接受这个调整
```

---

## 阶段 6：复盘归档

**目标**：分析实验结论，用户补充主观判断，共同形成复盘报告。

**交互模式**：AI 生成客观分析 → 用户补充主观感受和判断 → 共同确认结论 → 生成报告。

**退出条件**：用户确认复盘报告内容准确，后续计划已明确。

---

### Step 6-A：结果分析

```
【阶段6 · 结果分析】

实验目标：___
节点完成情况：[列出所有节点及状态：完成 / 跳过 / 失败]

结果汇总：
[粘贴参数记录表]

请帮我分析：
1. 各节点结果的横向对比：哪个方案表现最好？差距有多大？
2. 是否验证了最初的假设？
3. 有没有出乎意料的结果？

分析完后，逐条问我：
- 你是否认同这个分析？
- 有没有我遗漏的观察或你的主观判断想补充进来？
```

---

### Step 6-B：复盘报告生成

```
【阶段6 · 报告生成】

请结合我们的讨论，生成最终复盘报告，包含：
1. 实验结论（客观）
2. 与预期对比（哪些符合/超出/未达到预期）
3. 经验教训（做对了什么，下次可以改进什么）
4. 可复用资产（代码/数据/方法/模板）
5. 后续计划（下一批实验建议做什么）

报告生成后，问我是否有补充或修改，确认无误后存档。
```

## 快速使用指南

| 场景 | 触发词 |
|------|--------|
| 有了新想法 | `【阶段1 · 启动】我的想法是 ___` |
| 需求确认完毕 | `【阶段2 · 方案草稿】需求文档如下：___` |
| 方案写好了 | `【阶段3 · 评审】方案如下：___` |
| 评审通过 | `【阶段4 · 节点拆解】最终方案如下：___` |
| 开始某个节点 | `【阶段5 · 节点前确认】我准备开始节点 X` |
| 节点跑完了 | `【阶段5 · 结果记录】节点 X 结果是 ___` |
| 遇到报错 | `【阶段5 · 异常处理】节点 X 出问题了 ___` |
| 全部做完了 | `【阶段6 · 结果分析】结果汇总如下：___` |

---

## 标准文件结构

```
实验名称/
├── 需求文档.md          ← 阶段1产出
├── 实验方案.md          ← 阶段2-3产出（含评审意见）
├── 实验手册.md          ← 阶段4产出
├── 实验结果记录表.md    ← 阶段5实时填写
└── 复盘报告.md          ← 阶段6产出
```
